A Threshold Blind Credential Scheme for E-Cash

This page describes the cryptographic scheme implemented in ecash-ng-crypto: a Coconut-style blind credential built from Pointcheval–Sanders (PS) signatures over the pairing-friendly curve BLS12-381. An issuer blindly signs credentials over three hidden attributes; the user later shows the credential in zero knowledge, unlinkably to issuance.

Sections 1–5 present the complete scheme with a single issuer. Section 6 then explains how the issuer is replaced by a $t$-of-$n$ federation of guardians — a standard transformation that exploits only the linearity of signing in the secret key, and applies equally to ordinary short BLS signatures.

1. Setting and notation

Let $\mathbb{G}_1, \mathbb{G}_2, \mathbb{G}_T$ be the BLS12-381 pairing groups of prime order $p$, with bilinear map $e : \mathbb{G}_1 \times \mathbb{G}_2 \to \mathbb{G}_T$ and fixed generators $g_1 \in \mathbb{G}_1$, $g_2 \in \mathbb{G}_2$. All other generators are derived by hashing domain-separation strings to the curve (nothing-up-my-sleeve):

  • $G_p, H_p \in \mathbb{G}_1$ — Pedersen commitment basis for amounts,
  • $G, H_1, H_2, H_3 \in \mathbb{G}_1$ — basis for the attribute commitment.

Each e-cash note carries three attributes in $\mathbb{Z}_p$:

AttributeMeaning
$m_1$the amount (a 64-bit integer lifted to $\mathbb{Z}_p$)
$m_2$a uniformly random serial secret (basis of the nullifier)
$m_3 = \mathsf{H}_s(\mathit{auth})$an authentication hash, i.e. a spend condition fixed at issuance

2. Keys and the underlying signature

The issuer's secret key is a vector of four scalars $\mathsf{sk} = (x_0, x_1, x_2, x_3) \in \mathbb{Z}_p^4$ — one constant term plus one scalar per attribute. The public key publishes each component in both source groups:

$$\mathsf{pk} = \bigl(\, \{Y_j = x_j\, g_1\}_{j=0}^{3} \subset \mathbb{G}_1,\quad \{X_j = x_j\, g_2\}_{j=0}^{3} \subset \mathbb{G}_2 \,\bigr)$$

The $\mathbb{G}_1$ copies $Y_j$ are used by the client for unblinding; the $\mathbb{G}_2$ copies $X_j$ are used in pairing verification and in the spend proof.

Underlying primitive — Pointcheval–Sanders. A PS signature on $(m_1, m_2, m_3)$ with base point $h \in \mathbb{G}_1$ is the pair $(h, \sigma)$ where $$\sigma = \Bigl(x_0 + \textstyle\sum_{j=1}^{3} x_j m_j\Bigr)\, h,$$ verified by the single pairing equation $$e\!\left(h,\; X_0 + \textstyle\sum_{j=1}^{3} m_j X_j\right) \;=\; e(\sigma,\; g_2).$$ PS signatures are re-randomizable: for any $r \in \mathbb{Z}_p^\ast$, $(r\,h,\; r\,\sigma)$ is an equally valid, unlinkable signature on the same messages.

3. Issuance (blind signing)

The user samples randomness $r_p, r_m, r_1, r_2, r_3 \stackrel{\$}{\leftarrow} \mathbb{Z}_p$ and forms five group elements:

$$\begin{aligned} \mathsf{pc} &= m_1\, G_p + r_p\, H_p && \text{(Pedersen commitment to the amount)} \\[2pt] c_m &= m_1 H_1 + m_2 H_2 + m_3 H_3 + r_m\, G && \text{(commitment to all attributes)} \\[2pt] h &= \mathsf{H}_{\mathbb{G}_1}(c_m) && \text{(derived signature base)} \\[2pt] c_j &= m_j\, h + r_j\, G, \quad j \in \{1,2,3\} && \text{(blinded attributes)} \end{aligned}$$

Why $h = \mathsf{H}_{\mathbb{G}_1}(c_m)$? Deriving the base point from a binding commitment to the attributes is the Coconut trick: neither party can choose $h$ adversarially, every issuance gets a fresh base bound to the message vector, and the issuer can recompute $h$ itself from the public $c_m$.

3.1 The issuance proof

The well-formedness of $(\mathsf{pc}, c_m, c_1, c_2, c_3)$ is exactly the statement that this tuple lies in the image of the group homomorphism $\varphi_h : \mathbb{Z}_p^8 \to \mathbb{G}_1^5$, $$\varphi_h(m_1, m_2, m_3, r_p, r_m, r_1, r_2, r_3) \;=\; \begin{pmatrix} m_1\, G_p + r_p\, H_p \\[2pt] m_1 H_1 + m_2 H_2 + m_3 H_3 + r_m\, G \\[2pt] m_1\, h + r_1\, G \\[2pt] m_2\, h + r_2\, G \\[2pt] m_3\, h + r_3\, G \end{pmatrix} \;=\; \begin{pmatrix} \mathsf{pc} \\[2pt] c_m \\[2pt] c_1 \\[2pt] c_2 \\[2pt] c_3 \end{pmatrix},$$ and the user proves knowledge of a preimage with a generalized Schnorr $\Sigma$-protocol, made non-interactive by Fiat–Shamir (domain tag FEDIMINT_ECASH_CHALLENGE_ISSUANCE):

$$\vec{\rho} \stackrel{\$}{\leftarrow} \mathbb{Z}_p^8, \qquad \vec{R} = \varphi_h(\vec{\rho}), \qquad c = \mathsf{H}_s(\vec{Y} \,\|\, \vec{R}), \qquad \vec{s} = \vec{\rho} + c\,\vec{x},$$ $$\text{verify:}\quad \varphi_h(\vec{s}) \;\stackrel{?}{=}\; c\,\vec{Y} + \vec{R}, \qquad \text{where } \vec{Y} = (\mathsf{pc}, c_m, c_1, c_2, c_3).$$

Crucially, the proof ties the same $m_1$ into both $\mathsf{pc}$ and $c_1$: the amount inside the Pedersen commitment is the amount being signed. The issuer never learns $m_1, m_2, m_3$. (A range proof over $\mathsf{pc}$ can be attached externally by the transaction layer.)

3.2 Blind signing and unblinding

The issuer verifies the proof, recomputes $h = \mathsf{H}_{\mathbb{G}_1}(c_m)$, and returns the blinded signature

$$\widetilde{\sigma} \;=\; x_0\, h + \sum_{j=1}^{3} x_j\, c_j \;=\; \underbrace{\Bigl(x_0 + \textstyle\sum_j x_j m_j\Bigr) h}_{\text{PS signature } \sigma} \;+\; \underbrace{\sum_{j=1}^{3} r_j \,\bigl(x_j\, g_1\bigr)}_{\text{blinding term}}.$$

Because the blinding term is a known linear combination of the public key components in $\mathbb{G}_1$, the user removes it using public information only: $$\sigma = \widetilde{\sigma} - \sum_{j=1}^{3} r_j\, Y_{j},$$ and checks the result against the $X_j$ via the PS pairing equation.

3.3 Re-randomization

Finally the user samples $r \stackrel{\$}{\leftarrow} \mathbb{Z}_p^\ast$ and stores $$(h', \sigma') = (r\,h,\; r\,\sigma).$$ The issuer saw $h$ during issuance but never sees $h'$ — under DDH in $\mathbb{G}_1$ the shown pair is unlinkable to the issuance session.

4. Spend (showing the credential)

To spend, the user reveals $(h', \sigma')$, the authentication preimage data (so the verifier can recompute $m_3$), and two elements:

$$P = m_1\, G_p + r_p'\, H_p \;\in \mathbb{G}_1 \qquad\text{(fresh Pedersen commitment, new blinding } r_p'\text{)},$$ $$K = m_1\, X_1 + m_2\, X_2 \;\in \mathbb{G}_2 \qquad\text{(the hidden-attribute part of the verification equation)},$$

together with a second Fiat–Shamir $\Sigma$-proof (domain tag FEDIMINT_ECASH_CHALLENGE_SPEND) of knowledge of $(m_1, m_2, r_p')$ for the homomorphism $$\psi : \mathbb{Z}_p^3 \to \mathbb{G}_1 \times \mathbb{G}_2, \qquad \psi(m_1, m_2, r_p') = \bigl(\, m_1 G_p + r_p' H_p,\;\; m_1 X_1 + m_2 X_2 \,\bigr) = (P, K).$$

The verifier accepts iff the $\Sigma$-proof verifies and

$$e\!\left(h',\; X_0 + K + m_3\, X_3\right) \;=\; e(\sigma',\; g_2).$$

What each element achieves

ElementProperty
$P$ The amount never appears in the clear; the transaction layer can balance-check commitments homomorphically ($\sum P_{\text{in}} = \sum P_{\text{out}}$).
$K$ The nullifier: deterministic in $(m_1, m_2)$, so double-spending the same note reproduces the same $K$; the random $m_2$ makes $K$ unique per note and unlinkable to anything seen at issuance.
$m_3$ The spend condition: it is baked into $\sigma$, so only a spender who can produce the committed authentication data yields a verifying equation. Since the issuer only ever saw $m_3$ blinded, revealing it now does not link the spend to the issuance.
$(h', \sigma')$ A re-randomized PS pair — fresh group elements every spend, unlinkable to the issuance transcript.

5. Security properties (informal)

  • Unforgeability — from the unforgeability of Pointcheval–Sanders signatures (LRSW-style assumption) plus the soundness of the two $\Sigma$-proofs and the binding of the commitments; the hashed base $h = \mathsf{H}_{\mathbb{G}_1}(c_m)$ prevents base-point malleability across issuances.
  • Blindness — the issuer sees only Pedersen-style commitments $(\mathsf{pc}, c_m, c_1, c_2, c_3)$, all perfectly or computationally hiding, plus zero-knowledge proofs.
  • Unlinkability — re-randomization of $(h, \sigma)$ and fresh commitment randomness $r_p'$ make the spend transcript independent of the issuance transcript (under DDH).
  • Double-spend detection — equality of nullifiers $K$.

6. The threshold construction

Nothing above requires a single issuer. The blinded signing operation $$\widetilde{\sigma} \;=\; x_0\, h + \sum_{j=1}^{3} x_j\, c_j$$ is $\mathbb{Z}_p$-linear in the secret key $(x_0, \dots, x_3)$ — and any signature whose signing map is linear in the key can be thresholdized by Shamir secret sharing plus Lagrange interpolation in the exponent. This is exactly the standard construction for threshold BLS, where $\sigma = x \cdot \mathsf{H}(m)$ is linear in the single key scalar $x$; here it is applied component-wise to a four-scalar key.

6.1 Key sharing

Each $x_j$ is Shamir-shared: pick random polynomials $f_j \in \mathbb{Z}_p[X]$ of degree $t-1$ with $f_j(0) = x_j$, and give guardian $i \in \{1,\dots,n\}$ the key share $$\mathsf{sk}_i = \bigl(f_0(i),\, f_1(i),\, f_2(i),\, f_3(i)\bigr),$$ with corresponding public key shares $Y_{i,j} = f_j(i)\, g_1$ and $X_{i,j} = f_j(i)\, g_2$. The aggregate public key is unchanged: $Y_j = f_j(0)\, g_1$, $X_j = f_j(0)\, g_2$.

6.2 Share signing and aggregation

Each guardian runs the issuer's algorithm from §3.2 with its own key share: $$\widetilde{\sigma}_i \;=\; f_0(i)\, h + \sum_{j=1}^{3} f_j(i)\; c_j.$$ The user unblinds each share against the guardian's $\mathbb{G}_1$ key shares, $\sigma_i = \widetilde{\sigma}_i - \sum_j r_j\, Y_{i,j}$, and verifies it against the $X_{i,j}$ via the PS pairing equation — so misbehaving guardians are identified immediately. Any set $S$ of $t$ valid shares interpolates to the federation signature: $$\sigma = \sum_{i \in S} \lambda_i^S\, \sigma_i, \qquad \lambda_i^S = \prod_{\substack{j \in S \\ j \neq i}} \frac{j}{\,j - i\,}.$$ This is just polynomial interpolation at $0$ carried out on group elements: since $\sigma_i = \bigl(f_0(i) + \sum_j f_j(i)\, m_j\bigr) h$ and the map $x \mapsto x\,h$ is linear, the Lagrange combination evaluates the polynomial $f_0 + \sum_j m_j f_j$ at $0$, yielding exactly the single-issuer signature $\sigma = \bigl(x_0 + \sum_j x_j m_j\bigr) h$ of §2.

Everything downstream — re-randomization, spending, verification — is identical to the single-issuer scheme and verifies against the aggregate key. Fewer than $t$ guardians learn nothing about $\mathsf{sk}$ and cannot produce a valid share combination.

Implementation caveat (prototype). Hash-to-group is implemented by seeding a ChaCha RNG with SHA-256 rather than a standard hash_to_curve ciphersuite — fine for unknown-discrete-log generation, but non-interoperable.
